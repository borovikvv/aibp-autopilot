# Design: Issues #40, #42, #43 — unified storage, tiered experiment thresholds, content differentiation

**Date:** 2026-07-08 (rev. 2 — review comments incorporated)
**Issues:** #43 (SQLite→PostgreSQL), #42 (risk-tiered experiment thresholds), #40 (RU sources + weekly_case + competitor dedup)
**Related ADRs:** 0001, 0002, 0005, 0007, 0008, 0010

## Context

Three open issues, implemented together in one cycle because #43 (unify storage) is
infrastructure that #42 and #40 build on. Order of implementation: **#43 → #42 → #40**.

| Decision | Choice |
|---|---|
| #43 — migrate existing SQLite data? | No, **with a precondition** (see 1.0): prod VPS is not deployed yet, local DB is empty. Verified at implementation time, not assumed. |
| #43 — replacement style | Rewrite SQL inside existing functions on top of `db/connection.py` helpers; signatures unchanged. |
| #43 — test strategy | Unit tests mock the thin `db.py` layer; one integration pass against a real PG (`make test-integration`, skipped without `TEST_DATABASE_URL`). Plain `pytest` stays hermetic and fast. |
| #42 — scope | Full: risk-tiered thresholds + experiment-power dashboard section + historical simulation (with metric-change caveat). |
| #40 — competitor post collection | **Scraping the public web preview `t.me/s/<channel>`** (plain httpx + HTML parse). Optional Firecrawl API fallback when the direct fetch is blocked. NOT Bot API — a bot cannot read channels it is not a member of. |
| #40 — competitor dedup mechanism | Embeddings via OpenRouter `/api/v1/embeddings` (endpoint confirmed available), stored in pgvector. Grey-zone LLM check for cross-lingual pairs. |
| #40 — weekly_case format | New slot `weekly_case` in `policy.yaml`; replaces the morning post one weekday/week. Weekday check lives **inside `run()`** for both slots — single source of truth in policy. |

## Part 1 — #43: Consolidate self-learning storage into PostgreSQL

**Goal:** eliminate `data/self_learning.db` (SQLite); all self-learning tables live in
PostgreSQL alongside the content store; cross-store joins (engagement × post_features ×
link_clicks) become single-DB queries.

### 1.0 Precondition — verify there is no data to migrate

Issue #43's acceptance criteria include a data-transfer script. It is dropped **only if**
both hold at implementation time:

1. the prod VPS (docs/deploy_vps.md rollout) is not yet deployed, or its
   `data/self_learning.db` contains no `engagement_metrics` rows;
2. the local `data/self_learning.db` has no rows worth keeping.

Check: `sqlite3 data/self_learning.db "SELECT COUNT(*) FROM engagement_metrics"` locally
and on the VPS (if it exists). If either has data, write the one-off transfer script after
all (CSV export → COPY import, verify row counts) — losing accumulated engagement
snapshots would starve the reward function (#37) and the bandit.

### 1.1 New migration `0009_self_learning_tables.py`

Creates 8 tables in PG (port of the SQLite schema in `self_learning/db.py:SCHEMA`).
Dialect conversions:

| SQLite construct | PostgreSQL equivalent |
|---|---|
| `INTEGER PRIMARY KEY AUTOINCREMENT` | `bigserial PRIMARY KEY` |
| `INTEGER PRIMARY KEY` (feed_item_id) | `bigint PRIMARY KEY REFERENCES feed_items(id)` (real FK) |
| `TEXT` holding JSON | `jsonb` |
| composite `PRIMARY KEY (a, b)` | unchanged |
| `TIMESTAMP` (text ISO) | `timestamptz` |

Tables: `post_features`, `engagement_metrics`, `policies`, `experiments_log`,
`prompt_changes`, `bandit_arms`, `bandit_observations`, `autopilot_events`.

Indexes mirror the SQLite indexes (`idx_engagement_post`, etc.). The `_COLUMN_MIGRATIONS`
backfill (`assignment_mode`, `cta_variant`) is folded into the base CREATE — the migration
creates the final schema, so `_ensure_columns` is deleted.

### 1.2 Value-type fallout (the main source of migration bugs)

SQLite returned ISO **strings** and JSON **text**; psycopg2 returns `datetime` and `dict`.
Every consumer of these values changes behavior. Known sites — each must be fixed and is
part of this issue's review checklist:

| Site | Today | After PG |
|---|---|---|
| `reward.py: load_subscriber_series`, `compute_rewards_for_posts` | `datetime.fromisoformat(r["measured_at"])`, `fromisoformat(post["posted_at"])` | values are already `datetime` — drop parsing; keep a small `_as_dt()` shim only where inputs may still be ISO strings (dicts built in tests) |
| `decision_engine.py: make_decision` | `fromisoformat(experiment["started_at"])` | already `datetime` |
| `decision_engine.py: apply_promotion` | `json.loads(row["json_blob"])` | jsonb → already `dict`; `json.loads` removed |
| `engagement_collector.py` feature extraction | `policy_blob` written via `json.dumps` | write with `psycopg2.extras.Json` (or `%s::jsonb`) |
| `dashboard.py` DASHBOARD_TEMPLATE | Jinja slices strings: `exp.started_at[:19]`, `ev.event_at[:19]` | `datetime` is not subscriptable → template crashes. Format in Python (`.strftime`) or add a `fmt_ts` Jinja filter; audit **every** `[:N]` on a DB value in the template |
| all writers of jsonb columns | `json.dumps(...)` into TEXT | `Json(...)` adapter |
| comparisons `posted_at >= ?` with ISO-string params | string compare | pass `datetime` params; psycopg2 adapts them natively |

Conversion grep checklist (run before review):
`grep -rn "fromisoformat\|json.loads\|julianday\|datetime(\|date(\|INSERT OR" src/` and
`grep -n "\[:1[0-9]\]" src/aibp/observability/dashboard.py`.

### 1.3 Rewrite `self_learning/db.py`

`sqlite_conn()`, `import sqlite3`, `get_db_path()`, `_ensure_columns`,
`_COLUMN_MIGRATIONS`, and `init_db()` are **deleted**. The surviving functions are
rewritten on `db/connection.py` helpers (`fetch_all`, `fetch_one`, `execute`,
`execute_returning`, `db_conn`):

- `get_snapshot_at_horizon(feed_item_id, hours)` — the `julianday()` proximity sort becomes
  `ORDER BY ABS(EXTRACT(EPOCH FROM (em.measured_at - pf.posted_at)) / 3600.0 - %s) LIMIT 1`.
  Returns `dict | None` as before.
- `save_policy_version(...)` — `INSERT OR REPLACE` →
  `INSERT ... ON CONFLICT (version) DO UPDATE SET ...`.
- `log_autopilot_event(...)` — `?` → `%s`, via `execute()`; details through `Json`.
- `policy_version(policy_dict)` — unchanged (pure, no DB).
- `ENGAGEMENT_HORIZON_HOURS` — unchanged (constant).

`telegram_lock.py` loses its `get_db_path()` dependency — the lock file moves to
`data/telegram_getupdates.lock` via `PROJECT_ROOT` directly.

### 1.4 Callers

Files with direct `sqlite_conn()` usage (each block rewritten):

- `self_learning/decision_engine.py` — `get_ready_experiments`,
  `get_engagement_for_policy_version`, `apply_promotion`, `mark_pending_approval`,
  `reject_experiment`.
- `self_learning/engagement_collector.py` — feature insert, engagement insert,
  existing-features lookup.
- `self_learning/reward.py` — `load_subscriber_series`, `all_posted` query.
- `self_learning/pattern_miner.py`, `approvals.py`, `auto_rollback.py`, `bandit.py`
  (incl. `datetime('now', '-48 hours')` → `now() - interval '48 hours'`),
  `interleave.py`, `shadow_runner.py`.
- `monetization/offers.py` — `update_offer_outcomes` observation check.
- `observability/dashboard.py` — every `get_*` function.
- `growth/competitor_monitor.py` — `get_subscriber_series` (incl. `date(x)` → `x::date`),
  `_recent_auth_failures`, `get_our_er_percent`.

Rule: wherever a caller opened `sqlite_conn()` to run a one-off query, either route it
through a new helper in `db.py` (when reused) or inline it on `db_conn()`/`fetch_all`.
Placeholders `?` → `%s` everywhere.

### 1.5 Removals and doc updates

- `Settings.self_learning_db_path` field + `SELF_LEARNING_DB_PATH` from `.env.example`.
- `data/self_learning.db` (after 1.0 verification).
- `init_db()` — tables now created by migration `0009`. **Callers to update:**
  `scripts/bootstrap.py`, `cli.py db-init`, Makefile targets, and
  `prompts/hermes_bootstrap.md` if any of them invoke it.
- Docs that state "SQLite for self-learning": README architecture diagram/text,
  `docs/install.md`, `docs/deploy_vps.md`, ADR-0001 (add a superseded-note), ADR-0006
  (its "sync psycopg2" reasoning now covers self-learning too).

### 1.6 Tests

- Pure-function tests (`compute_decision`, `compute_rewards_for_posts`, bandit math,
  reward math) — unaffected, no DB.
- **Unit tests stay hermetic**: the `db.py` surface is small (`get_snapshot_at_horizon`,
  `save_policy_version`, `log_autopilot_event` + `fetch_all/execute` call sites), so
  DB-touching unit tests mock these helpers (`unittest.mock.patch`), replacing today's
  implicit tmp-SQLite fixtures. No live PG needed for `pytest`.
- **One integration pass against real PG**: `tests/integration/test_self_learning_cycle.py`
  runs against `TEST_DATABASE_URL` (creates schema via migrations, exercises the full
  cycle), `pytest.mark.skipif` without the env var. New Makefile target
  `test-integration` documents how to run it (docker `postgres:16` one-liner in the
  target's comment).
- Acceptance: `pytest` green without PG; integration green with PG; dashboard renders;
  `python -m aibp.db.migrate --status` shows `0009` applied.

## Part 2 — #42: Risk-tiered experiment thresholds + power + simulation

**Goal:** experiments actually promote; the data-starvation problem is visible on the
dashboard; a simulation validates the new thresholds against history.

### 2.1 Tiered thresholds in `policy.yaml`

Tiers map to the existing risk classification in `safety.py` (`approval_required_for`):
low-risk = auto-applied with auto-rollback; high-risk = human approval gate.

```yaml
safety:
  experiment_tiers:
    low_risk:
      experiment_types: [rubric_weight, cta, source_score, visual]
      promote_probability: 0.90   # was 0.95; low cost of error, auto-rollback present
      min_effect_pct: 5
      experiment_window_days: 14
    high_risk:
      experiment_types: [post_param, regex_gate]
      promote_probability: 0.95
      min_effect_pct: 5
      experiment_window_days: 28   # extended: 21–28 from the issue
```

The flat `promote_probability` / `min_effect_pct` / `experiment_window_days` fields are
**removed from the file** — one source of truth. The **code** keeps a fallback to flat
fields because old policy versions stored in the `policies` table still contain them
(tier resolution must work when replaying/rolling back to an old version).

### 2.2 Tier resolution

New `self_learning/tiers.py` module:

```python
def load_tier_config(experiment_type: str, policy: dict | None = None) -> dict:
    """Return {promote_probability, min_effect_pct, experiment_window_days} for the type.

    Looks up experiment_type in safety.experiment_tiers[*].experiment_types.
    Defaults to high_risk values when the type is unknown (safer).
    Falls back to flat safety.* fields when experiment_tiers is absent
    (old policy versions stored in the policies table).
    """
```

Unknown type → high_risk tier (conservative). This is unit-tested.

### 2.3 decision_engine wiring

- `get_ready_experiments()` — the readiness cutoff becomes per-experiment: each
  experiment's `started_at < now - its_tier.experiment_window_days`, not a single global
  cutoff. Load tier per row via `experiment_type`.
- `make_decision(experiment)` — loads tier config for `experiment["experiment_type"]`,
  passes `promote_probability`, `min_effect`, `give_up_days = window + 7` into
  `compute_decision(...)`.
- `compute_decision(...)` — **unchanged**. It already accepts thresholds as parameters;
  existing tests stay green.

### 2.4 Experiment-power dashboard section

New `get_experiment_power()` in `dashboard.py` + a new `<h2>Experiment Power</h2>` section
in `DASHBOARD_TEMPLATE`. Per active (`shadow_running`) experiment:

| Field | Source |
|---|---|
| control n / shadow n | counts from `post_features` per policy_version |
| target n | `tier.experiment_window_days × posts_per_day` (posts_per_day from recent history, default 2) |
| days_to_decision | `tier.experiment_window_days − exp_age_days` |
| current P(shadow>control) | if n≥5 in both groups: run `compute_decision` on current data; else `—` |
| status | `ready_to_decide` / `on_track` / `behind` (n growth slower than post pace) |

This is practical power visibility — "how much data do we have vs how much we need until
the decision window" — directly answering the issue's "show progress/forecast to decision".

### 2.5 Historical simulation

New `scripts/simulate_thresholds.py` standalone script:

1. Load all finished experiments (`promoted`/`rejected`/`rolled_back`) from `experiments_log`.
2. `experiments_log` stores only aggregates (mean, n) — `compute_decision` needs raw
   per-post rates, so **recompute** them via
   `get_engagement_for_policy_version(policy_before/after, since=started_at)` +
   `compute_reward_rates`.
3. For each experiment, run `compute_decision` under (a) old flat thresholds (0.95/5%/14d)
   and (b) new tiered thresholds; print a comparison table.
4. **Caveat printed in the output**: recomputed rates use the composite reward (#37),
   while historical decisions were made on views/subs — the table compares thresholds on
   today's metric, not a literal replay of past decisions. `rolled_back` experiments are
   listed separately (they were promoted and then failed — a threshold that re-promotes
   them is not a win).
5. If finished-experiment count is small, report "insufficient history for simulation"
   honestly — do not fabricate.

### 2.6 Tests

- `compute_decision` existing tests — green (parameterized thresholds).
- New: `test_load_tier_config` — type→tier mapping, unknown-type→high_risk, old-policy
  flat-field fallback.
- New: `test_get_experiment_power` — shape of returned data, status classification.
- New: `test_simulate_thresholds` — smoke test on empty/small sample (does not crash,
  prints the caveat).

## Part 3 — #40: RU sources + weekly_case slot + competitor dedup by embeddings

**Goal:** differentiate from competitors — new source material, a signature format, and
dedup against competitor posts.

### 3.1 New sources in `config/rss_feeds.yaml`

Add ≥5 sources across `research` / `cases` / `regulation` / `ru`. Extend the `category`
value set (currently `news | blog | research | company`) with `cases` and `regulation` so
enrichment can route them. `rss_collector.py` already reads `category`/`lang` — no code
change; `load_rss_feeds_to_db()` inserts them unchanged.

Candidate feeds — **verified by `make check-feeds` before the commit lands, not "during
implementation"**:

- HuggingFace Daily Papers — `research`, en (replaces Papers with Code, which shut down
  in 2025);
- Stanford HAI news / MIT Sloan Management Review AI — `cases`, en (pick whichever has a
  working RSS; McKinsey has no stable feed);
- EU AI Act / European Commission digital-strategy news — `regulation`, en;
- VC.ru «Технологии» — `news`, ru;
- Forbes RU — `cases`, ru;
- Habr AI hub weight 0.8 → 1.0.

New `make check-feeds` target → small script: fetch every feed URL, assert HTTP 200 +
parseable XML + ≥1 entry; prints a table. Run in review; keeps AC "≥5 new sources" honest.

### 3.2 New slot `weekly_case`

**`policy.yaml`** — new `post_params.weekly_case` block + top-level config:
```yaml
post_params:
  weekly_case:
    target_chars: [1200, 1800]
    paragraphs: [4, 6]
    max_bold: 4
    max_emoji: 0
    scheduled_hour_msk: 10   # replaces the morning post on case day
weekly_case:
  enabled: true
  weekday: 2                 # 0=Mon; Wednesday replaces morning with the case
```

**Day mechanics — single source of truth, checked inside `run()`** (no cron-side day
logic, so the Hermes cron schedule never duplicates the weekday):

- both cron jobs fire daily at 10:00 MSK: the existing morning job and a new
  `generate --slot weekly_case` job;
- `run(slot="morning")`: if `weekly_case.enabled and today.weekday() == weekly_case.weekday`
  → log `morning_replaced_by_weekly_case`, return 0;
- `run(slot="weekly_case")`: if not (`enabled` and today matches) → log and return 0.

Exactly one of the two produces a post on any given day. Registration of the new cron job
goes into `prompts/hermes_bootstrap.md` (README: cron changes require re-registering).

**`cli.py`** — `generate --slot` gains `weekly_case` in its `click.Choice`.

**`generation/pipeline.py`:**
- `select_candidate(slot="weekly_case", ...)` — prefer candidates with
  `category IN (cases, regulation, research)` and `source_fit_score >= 4`; fall back to
  the normal ranked list if none qualify.
- New `WEEKLY_CASE_PROMPT` Jinja2 template — the signature format: process → tool →
  before/after numbers → stop condition. One source, one deep breakdown, metrics required.
- `GENERATION_PROMPT` gains an `{% elif slot == "weekly_case" %}` guidance branch.

**`enrichment/pipeline.py`:**
- `recommended_format` gains value `"weekly_case"`. The enrichment prompt is updated: for
  sources of category `cases`/`regulation` containing implementation figures, set
  `recommended_format: weekly_case`.

**`generation/quality_gate.py`:**
- New gate `case_metrics_presence` for `slot == "weekly_case"`: the post must contain at
  least one numeric measurement (before/after, %, time, cost). On failure → `warn` →
  informed retry (same mechanism as the existing `metric_presence` gate for morning).

### 3.3 Competitor dedup by embeddings

**Collecting competitor posts — public web preview scraping.** A bot cannot read channels
it is not a member of, so Bot API is out. Public channels expose their recent posts at
`https://t.me/s/<channel>` — no token, no login.

New `growth/competitor_scraper.py`:

```python
def fetch_channel_posts(username: str, limit: int = 20) -> list[dict]:
    """Recent posts of a public channel via the t.me/s/ web preview.

    1. Primary: httpx GET https://t.me/s/{username} with a browser-like UA;
       parse `.tgme_widget_message` blocks (message id, datetime attr, text).
    2. Fallback (optional): Firecrawl API when FIRECRAWL_API_KEY is set and the
       direct fetch fails or returns no messages (datacenter-IP blocks) —
       POST https://api.firecrawl.dev/v2/scrape with the same URL, parse the
       returned HTML/markdown identically.
    3. Both failed → [] with a warning. Weekly cron; a missed week only delays
       dedup data, nothing user-facing breaks.
    """
```

Parsing is stdlib (`html.parser` or regex over the stable `tgme_widget_message_text`
markup) — no new hard dependency; Firecrawl is config-gated, not required.

Wire into `growth/competitor_monitor.py` weekly cron: for each channel in
`config/competitors.yaml`, fetch posts, skip already-stored `(channel, message_id)`,
embed new texts (batch), insert into `competitor_posts`.

**New PG table** (migration `0010_competitor_posts.py`, after #43 so already in PG):
```sql
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE IF NOT EXISTS competitor_posts (
    id            bigserial PRIMARY KEY,
    channel       text NOT NULL,
    message_id    text,
    posted_at     timestamptz,
    title         text,
    text_excerpt  text,
    embedding     vector(1536),
    fetched_at    timestamptz DEFAULT now(),
    UNIQUE (channel, message_id)
);
```

No ivfflat index: at this table's scale (hundreds–thousands of rows) exact scan with the
`<=>` operator is fast and correct; ivfflat on a near-empty table is noise. Revisit if
the table ever exceeds ~100k rows.

**Ops:** `CREATE EXTENSION vector` requires the `postgresql-XX-pgvector` package and a
superuser. Add installation to `scripts/setup_postgres.sh` and a step to
`docs/deploy_vps.md`, otherwise migration `0010` fails on a clean VPS.

**Embeddings via OpenRouter** — endpoint confirmed available
(`POST /api/v1/embeddings`, OpenAI-compatible). New
`OpenRouterClient.embed(texts: list[str]) -> list[list[float]]` in
`enrichment/llm_client.py`; model from `OPENROUTER_EMBEDDING_MODEL` (default
`openai/text-embedding-3-small`), dim 1536; cost logged like `chat()`.

**Pre-generation check** — new `self_learning/competitor_dedup.py`:

```python
def check_duplicate(title: str, text: str, policy: dict) -> str:
    """'duplicate' | 'unique' | 'grey' for a candidate vs competitor_posts.

    Cosine similarity via pgvector `<=>`:
      >= dup_threshold (default 0.85)  → 'duplicate'
      <= grey_threshold (default 0.70) → 'unique'
      in between                        → 'grey'
    Degrades to 'unique' (no block) on pgvector/OpenRouter unavailability —
    publishing a post is better than skipping it.
    """
```

Cross-lingual caveat: competitors post in Russian about the same English news;
`text-embedding-3-small` gives noticeably lower cross-lingual similarity, so a hard 0.85
misses real duplicates. Hence the grey zone: for `'grey'` results a small LLM
(`chat_json`, cheap model) answers "is this the same news story?" — its verdict decides.
Both thresholds configurable via `policy.competitor_dedup.{dup_threshold,grey_threshold}`;
calibrate against live data after 2–3 weeks of collected competitor posts (log every
similarity score for later analysis).

**Wiring in `generation/pipeline.run()`** — after `select_candidate()`, before
`generate_post()`. `select_candidate` gains an `exclude_ids: set[int] | None` parameter;
`run()` loops: candidate → dedup check → on duplicate add to `exclude_ids`, log
`candidate_skipped_competitor_dup`, re-select (max 3 attempts) → no candidate left →
skip the slot (return 0).

### 3.4 Tests

- `test_weekly_case_prompt` — template renders with the case structure.
- `test_weekly_case_day_mechanics` — morning run skips on case day; weekly_case run
  skips on other days; exactly one fires per day across the week.
- `test_weekly_case_quality_gate` — `case_metrics_presence` rejects a metric-less post,
  passes one with numbers.
- `test_competitor_scraper` — parses a saved t.me/s/ HTML fixture into posts; empty/blocked
  page → []; Firecrawl fallback invoked only when configured and primary fails (mocked).
- `test_competitor_dedup` — duplicate / unique / grey classification on stubbed
  similarities; grey resolved by mocked LLM verdict; degrades to 'unique' when embedding
  API / pgvector unavailable.
- `test_embed` (smoke) — `OpenRouterClient.embed` shapes the request correctly (mocked).
- `test_select_candidate_exclude_ids` — excluded candidate is not re-selected.
