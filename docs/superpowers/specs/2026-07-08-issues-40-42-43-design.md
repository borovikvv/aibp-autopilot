# Design: Issues #40, #42, #43 тАФ unified storage, tiered experiment thresholds, content differentiation

**Date:** 2026-07-08
**Issues:** #43 (SQLiteтЖТPostgreSQL), #42 (risk-tiered experiment thresholds), #40 (RU sources + weekly_case + competitor dedup)
**Related ADRs:** 0001, 0002, 0007, 0008, 0010

## Context

Three open issues, implemented together in one cycle because #43 (unify storage) is
infrastructure that #42 and #40 build on. Order of implementation: **#43 тЖТ #42 тЖТ #40**.

| Decision (from brainstorming) | Choice |
|---|---|
| #43 тАФ migrate existing SQLite data? | No. Local `data/self_learning.db` is empty; create tables in PG, switch code. No data-migration script. |
| #42 тАФ scope | Full: risk-tiered thresholds + experiment-power dashboard section + historical simulation. |
| #40 тАФ competitor dedup mechanism | Embeddings via OpenRouter (`text-embedding-3-small`), stored in pgvector. |

| #43 тАФ replacement style | Approach 1: rewrite SQL inside existing functions on top of `db/connection.py` helpers; signatures unchanged. |
| Embedding storage | pgvector (`CREATE EXTENSION vector`). |

## Part 1 тАФ #43: Consolidate self-learning storage into PostgreSQL

**Goal:** eliminate `data/self_learning.db` (SQLite); all self-learning tables live in
PostgreSQL alongside the content store; cross-store joins (engagement ├Ч post_features ├Ч
link_clicks) become single-DB queries.

### 1.1 New migration `0009_self_learning_tables.py`

Creates 7 tables in PG (port of the SQLite schema in `self_learning/db.py:SCHEMA`).
Dialect conversions:

| SQLite construct | PostgreSQL equivalent |
|---|---|
| `INTEGER PRIMARY KEY AUTOINCREMENT` | `bigserial PRIMARY KEY` |
| `INTEGER PRIMARY KEY` (feed_item_id) | `bigint REFERENCES feed_items(id)` (real FK) |
| `TEXT` holding JSON | `jsonb` |
| composite `PRIMARY KEY (a, b)` | unchanged |
| `TIMESTAMP` (text ISO) | `timestamptz` |

Tables: `post_features`, `engagement_metrics`, `policies`, `experiments_log`,
`prompt_changes`, `bandit_arms`, `bandit_observations`, `autopilot_events`.

Indexes mirror the SQLite indexes (`idx_engagement_post`, etc.). The `_COLUMN_MIGRATIONS`
backfill (`assignment_mode`, `cta_variant`) is folded into the base CREATE тАФ the migration
creates the final schema, so `_ensure_columns` is deleted.

### 1.2 Rewrite `self_learning/db.py`

`sqlite_conn()`, `import sqlite3`, `get_db_path()`, `_ensure_columns`,
`_COLUMN_MIGRATIONS`, and `init_db()` are **deleted**. The surviving functions are
rewritten on `db/connection.py` helpers (`fetch_all`, `fetch_one`, `execute`,
`execute_returning`, `db_conn`):

- `get_snapshot_at_horizon(feed_item_id, hours)` тАФ the `julianday()` proximity sort becomes
  `ORDER BY ABS(EXTRACT(EPOCH FROM (em.measured_at - pf.posted_at)) / 3600.0 - %s) LIMIT 1`.
  Returns `dict | None` as before.
- `save_policy_version(...)` тАФ `INSERT OR REPLACE` тЖТ
  `INSERT ... ON CONFLICT (version) DO UPDATE SET ...`.
- `log_autopilot_event(...)` тАФ `?` тЖТ `%s`, via `execute()`.
- `policy_version(policy_dict)` тАФ unchanged (pure, no DB).
- `ENGAGEMENT_HORIZON_HOURS` тАФ unchanged (constant).

### 1.3 Callers

Files with direct `sqlite_conn()` usage (each block rewritten):

- `self_learning/decision_engine.py` тАФ `get_ready_experiments`,
  `get_engagement_for_policy_version`, `apply_promotion`, `mark_pending_approval`,
  `reject_experiment`. `?` тЖТ `%s`; `INSERT ... RETURNING` already PG-native.
- `self_learning/engagement_collector.py` тАФ `extract_features_for_post` insert,
  engagement insert, existing-features lookup.
- `self_learning/reward.py` тАФ `load_subscriber_series`, `all_posted` query.
- `self_learning/pattern_miner.py`, `approvals.py`, `auto_rollback.py`, `bandit.py`,
  `interleave.py`, `shadow_runner.py` тАФ same treatment.
- `observability/dashboard.py` тАФ every `get_*` function.
- `growth/competitor_monitor.py` тАФ `get_subscriber_series`, `_recent_auth_failures`,
  `get_our_er_percent`.

Rule: wherever a caller opened `sqlite_conn()` to run a one-off query, either route it
through a new helper in `db.py` (when reused) or inline it on `db_conn()`/`fetch_all`.
Placeholders `?` тЖТ `%s` everywhere. SQLite functions `date(x)` тЖТ `x::date`,
`INSERT OR REPLACE` тЖТ `ON CONFLICT`.

### 1.4 Removals

- `Settings.self_learning_db_path` field + `SELF_LEARNING_DB_PATH` from `.env.example`.
- `data/self_learning.db` (empty, gitignored content).
- `self_learning/db.py:init_db()` тАФ tables now created by migration `0009`.

### 1.5 Tests

- Pure-function tests (`compute_decision`, `compute_rewards_for_posts`, bandit math) are
  unaffected тАФ no DB.
- DB-touching tests currently run against SQLite implicitly. Add a `conftest.py` fixture
  providing a throwaway PG database (via env `TEST_DATABASE_URL`; CI creates the DB).
  `tests/integration/test_self_learning_cycle.py` is adapted to PG.
- Acceptance: all tests green; dashboard renders; a manual `python -m aibp.db.migrate
  --status` shows `0009` applied.

## Part 2 тАФ #42: Risk-tiered experiment thresholds + power + simulation

**Goal:** experiments actually promote; the data-starvation problem is visible on the
dashboard; a simulation validates the new thresholds against history.

### 2.1 Tiered thresholds in `policy.yaml`

Replace the three flat fields under `safety` with a tiered structure. Tiers map to the
existing risk classification in `safety.py` (`approval_required_for`):
low-risk = auto-applied with auto-rollback; high-risk = human approval gate.

```yaml
safety:
  experiment_tiers:
    low_risk:
      experiment_types: [rubric_weight, cta, source_score, visual]
      promote_probability: 0.90   # was 0.95; low cost of error, auto-rollback present
      min_effect_pct: 5
      experiment_window_days: 14
    high_risk:
      experiment_types: [post_param, regex_gate]
      promote_probability: 0.95
      min_effect_pct: 5
      experiment_window_days: 28   # extended: 21тАУ28 from the issue
  # Fallback for policies without experiment_tiers (backward compat):
  promote_probability: 0.95
  min_effect_pct: 5
  experiment_window_days: 14
```

### 2.2 Tier resolution

New `self_learning/tiers.py` module:

```python
def load_tier_config(experiment_type: str, policy: dict | None = None) -> dict:
    """Return {promote_probability, min_effect_pct, experiment_window_days} for the type.

    Looks up experiment_type in safety.experiment_tiers[*].experiment_types.
    Defaults to high_risk values when the type is unknown (safer).
    Falls back to flat safety.* fields when experiment_tiers is absent (old policy).
    """
```

Unknown type тЖТ high_risk tier (conservative). This is unit-tested.

### 2.3 decision_engine wiring

- `get_ready_experiments()` тАФ the readiness cutoff becomes per-experiment: each
  experiment's `started_at < now - its_tier.experiment_window_days`, not a single global
  cutoff. Load tier per row via `experiment_type`.
- `make_decision(experiment)` тАФ loads tier config for `experiment["experiment_type"]`,
  passes `promote_probability`, `min_effect`, `give_up_days = window + 7` into
  `compute_decision(...)`.
- `compute_decision(...)` тАФ **unchanged**. It already accepts thresholds as parameters;
  existing tests stay green.

### 2.4 Experiment-power dashboard section

New `get_experiment_power()` in `dashboard.py` + a new `<h2>Experiment Power</h2>` section
in `DASHBOARD_TEMPLATE`. Per active (`shadow_running`) experiment:

| Field | Source |
|---|---|
| control n / shadow n | counts from `post_features` per policy_version |
| target n | `tier.experiment_window_days ├Ч posts_per_day` (posts_per_day from recent history, default 2) |
| days_to_decision | `tier.experiment_window_days тИТ exp_age_days` |
| current P(shadow>control) | if nтЙе5 in both groups: run `compute_decision` on current data; else `тАФ` |
| status | `ready_to_decide` / `on_track` / `behind` (n growth slower than post pace) |

This is practical power visibility тАФ "how much data do we have vs how much we need until
the decision window" тАФ directly answering the issue's "show progress/forecast to decision".

### 2.5 Historical simulation

New `scripts/simulate_thresholds.py` standalone script (invoked as
`python scripts/simulate_thresholds.py`):

1. Load all finished experiments (`promoted`/`rejected`/`rolled_back`) from `experiments_log`.
2. For each, recompute `compute_decision` under (a) old flat thresholds (0.95/5%/14d) and
   (b) new tiered thresholds (0.90 for low-risk).
3. Print a comparison table: decisions-would-be-promoted count under old vs new.
4. If finished-experiment count is small (channel is young), report "insufficient history
   for simulation" honestly тАФ do not fabricate.

Closes the "simulation on historical data" acceptance criterion.

### 2.6 Tests

- `compute_decision` existing tests тАФ green (parameterized thresholds).
- New: `test_load_tier_config` тАФ typeтЖТtier mapping, unknown-typeтЖТhigh_risk, old-policy
  fallback.
- New: `test_get_experiment_power` тАФ shape of returned data, status classification.
- New: `test_simulate_thresholds` тАФ smoke test on empty/small sample (does not crash).

## Part 3 тАФ #40: RU sources + weekly_case slot + competitor dedup by embeddings

**Goal:** differentiate from competitors тАФ new source material, a signature format, and
dedup against competitor posts.

### 3.1 New sources in `config/rss_feeds.yaml`

Add тЙе5 sources across `research` / `cases` / `regulation` / `ru`. Extend the `category`
value set (currently `news | blog | research | company`) with `cases` and `regulation` so
enrichment can route them. `rss_collector.py` already reads `category`/`lang` тАФ no code
change; `load_rss_feeds_to_db()` inserts them unchanged.

New feeds (final URLs verified during implementation):
- Papers with Code (AI) тАФ `research`, en
- McKinsey AI Insights тАФ `cases`, en
- EU AI Act feed (EU Commission) тАФ `regulation`, en
- VC.ru ╨в╨╡╤Е╨╜╨╛╨╗╨╛╨│╨╕╨╕ тАФ `news`, ru
- Forbes RU тАФ `cases`, ru
- Raise Habr AI weight 0.8 тЖТ 1.0.

### 3.2 New slot `weekly_case`

**`policy.yaml`** тАФ new `post_params.weekly_case` block:
```yaml
post_params:
  weekly_case:
    target_chars: [1200, 1800]
    paragraphs: [4, 6]
    max_bold: 4
    max_emoji: 0
    scheduled_hour_msk: 10   # replaces the morning post on case day
weekly_case:
  enabled: true
  weekday: 2                 # 0=Mon; Wednesday replaces morning with the case
```

**`generation/pipeline.py`:**
- `select_candidate(slot="weekly_case", ...)` тАФ prefer candidates with
  `category IN (cases, regulation, research)` and `source_fit_score >= 4`; fall back to
  the normal ranked list if none qualify.
- New `WEEKLY_CASE_PROMPT` Jinja2 template тАФ the signature format: process тЖТ tool тЖТ
  before/after numbers тЖТ stop condition. One source, one deep breakdown, metrics required.
- `GENERATION_PROMPT` gains an `{% elif slot == "weekly_case" %}` guidance branch.
- `run(slot="weekly_case")` тАФ when `datetime.today().weekday() == weekly_case.weekday`,
  the case runs instead of morning. The day-check lives at the call site (Makefile/cron
  invokes `run(slot="weekly_case")` on the configured weekday).

**`enrichment/pipeline.py`:**
- `recommended_format` gains value `"weekly_case"`. The enrichment prompt is updated: for
  sources of category `cases`/`regulation` containing implementation figures, set
  `recommended_format: weekly_case`.

**`generation/quality_gate.py`:**
- New gate `case_metrics_presence` for `slot == "weekly_case"`: the post must contain at
  least one numeric measurement (before/after, %, time, cost). On failure тЖТ `warn` тЖТ
  informed retry (same mechanism as the existing `metric_presence` gate for morning).

### 3.3 Competitor dedup by embeddings

**New PG table** (migration `0010_competitor_posts.py`, after #43 so already in PG;
requires `CREATE EXTENSION vector`):
```sql
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE IF NOT EXISTS competitor_posts (
    id            bigserial PRIMARY KEY,
    channel       text NOT NULL,
    message_id    text,
    posted_at     timestamptz,
    title         text,
    text_excerpt  text,
    embedding     vector(1536),
    fetched_at    timestamptz DEFAULT now()
);
CREATE INDEX IF NOT EXISTS idx_competitor_posts_embedding
    ON competitor_posts USING ivfflat (embedding vector_cosine_ops);
```

**Collecting competitor posts** тАФ extend `growth/competitor_monitor.py` (weekly cron): for
each competitor, fetch recent posts via Telegram Bot API (`copyMessage` into the metrics
chat, read text, delete тАФ same pattern `engagement_collector` already uses), store title +
text + embedding in `competitor_posts`. TGStat gives stats only, not post text, so post
text comes from the Bot API.

**Embeddings via OpenRouter** тАФ new `OpenRouterClient.embed(texts: list[str]) -> list[list[float]]`
method in `enrichment/llm_client.py`: POST `/api/v1/embeddings`, model from
`OPENROUTER_EMBEDDING_MODEL` (default `openai/text-embedding-3-small`), dim 1536.

**Pre-generation check** тАФ new `self_learning/competitor_dedup.py`:
```python
def is_duplicate_of_competitor(title: str, text: str, threshold: float = 0.85) -> bool:
    """True when a competitor post is semantically near this candidate.

    Computes the query embedding, searches competitor_posts by cosine similarity
    (pgvector <=> operator). Degrades to False (no block) on pgvector/OpenRouter
    unavailability тАФ publishing a post is better than skipping it.
    """
```
Called in `generation/pipeline.run()` after `select_candidate()` and before
`generate_post()`. On a duplicate: log and re-rank to the next candidate (the same
`select_candidate` pool minus the duplicate); if no candidate remains, skip the slot.
Threshold configurable via `policy.competitor_dedup.threshold` (default 0.85).

### 3.4 Tests

- `test_weekly_case_prompt` тАФ template renders with the case structure.
- `test_weekly_case_quality_gate` тАФ `case_metrics_presence` rejects a metric-less post,
  passes one with numbers.
- `test_competitor_dedup` тАФ `is_duplicate_of_competitor` True on near-duplicate, False on
  unrelated; returns False (no crash) when the embedding API / pgvector is unavailable.
- `test_embed` (smoke) тАФ `OpenRouterClient.embed` shapes the request correctly (mocked).
