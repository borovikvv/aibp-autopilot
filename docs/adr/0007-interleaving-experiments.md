# ADR-0007: Interleaving вместо cross-channel shadow-сравнения

**Status:** Accepted
**Date:** 2026-07-06
**Supersedes:** часть ADR-0001 (shadow-тестирование через тестовый канал)

## Контекст

ADR-0001 закрепил схему shadow-тестирования: control-посты публикуются в
основной канал (`main`), shadow-посты с новой политикой — в тестовый канал
(`test`). Decision engine сравнивал engagement rate (views/subscribers) двух
групп через Welch's t-test.

Проблема: у тестового канала другая — почти нулевая — аудитория. Разница
engagement rate между каналами отражает разницу аудиторий, а не разницу
политик. Statistical test находит «значимые» различия, которые являются
артефактом схемы эксперимента. Self-learning обучается на шуме, который
выглядит как сигнал.

## Решение

1. **Отказ от cross-channel сравнения.** Decision engine больше не сравнивает
   посты каналов `main` и `test`. Функция
   `get_engagement_for_policy_version` фильтрует только `target_channel='main'`
   и только посты, опубликованные после старта эксперимента.

2. **Interleaving по дням в основном канале.** Активный эксперимент чередует
   политики по чётности дня (day-of-year, MSK): чётные дни — control
   (`policy_before`, текущий `config/policy.yaml`), нечётные дни — variant
   (`policy_after` из таблицы `policies` в PostgreSQL). Одна аудитория → сравнение
   статистически корректно. Назначение детерминировано датой, поэтому не
   требует дополнительного состояния и переживает рестарты.

3. **`experiments_log.assignment_mode`.** Новое поле со значениями:
   - `interleave` (default) — чередование политик в `main`;
   - `cross_channel` — legacy-записи старой схемы (решения по ним не
     принимаются, только исторические данные).

4. **Тестовый канал остаётся только как quality gate.** Stage-пайплайн
   (`pipeline_env='stage'` → канал `test`) продолжает работать для визуального
   контроля качества генерации под новой политикой, но его engagement-данные
   исключены из статистических решений.

## Две «политики», которые НЕЛЬЗЯ путать (issue #22)

После этого ADR в проекте есть два разных объекта, оба исторически звались
«shadow policy». Это разные вещи с разным назначением:

| | **Preview policy** | **Interleave variant** |
|---|---|---|
| Файл/источник | `config/policy.stage.yaml` (генерируемый) | PostgreSQL `policies`, версия `policy_after` |
| Пишет | `shadow_runner.apply_policy_to_stage()` | `policy_updater.create_experiment()` |
| Читает | `generation/pipeline` при `pipeline_env='stage'` | `self_learning/interleave.resolve_policy_for_today()` |
| Канал | `test` (превью) | `main` (прод, нечётные дни) |
| Назначение | **визуальный QA для человека** | **статистический тест** |
| Влияет на решения | ❌ нет | ✅ да |

Правило: `policy.stage.yaml` — это **preview для человека**, а не вариант
эксперимента. Статистику определяет только interleave-variant из PostgreSQL.

Про rename: переименование значения `pipeline_env='stage' → 'preview'` (как
предлагалось в issue #22) сознательно **не** сделано — это значение
персистится в `feed_items.pipeline_env`, зашито в `schema.sql`, миграции и
`select_candidate`; массовый rename ради нейминга потребовал бы миграции
данных и дал бы churn без функционального выигрыша. Устранение путаницы
достигается этой таблицей и комментариями в коде.

## Последствия

- Все посты канала `main` во время эксперимента несут в `post_features.policy_version`
  фактически применённую политику — control- и variant-группы выделяются по ней.
- Скорость набора данных не падает: те же 2 поста/день, но обе группы получают
  реальную аудиторию (раньше shadow-группа набирала мусорные данные).
- Риск: variant-политика в нечётные дни публикуется в прод. Смягчение: quality
  gate на stage до старта interleave + существующие rollback/anomaly guards.
