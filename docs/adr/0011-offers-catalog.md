# ADR-0011: Каталог офферов и выбор по ожидаемому доходу

**Status:** Accepted
**Date:** 2026-07-07
**Related:** ADR-0010 (composite reward), issue #38, issue #15/#16

## Контекст

CTA-вариант `affiliate_link` был фразой «по ссылке ниже», указывающей на
ссылку источника. Реальных офферов и представления дохода в системе не было —
«монетизационная воронка» не содержала денег, оптимизировать было нечего.

## Решение

1. **Каталог `offers` в PostgreSQL** (миграция 0007): slug, title, target_url,
   `topics text[]` (теги из словаря topic_cluster enrichment-слоя; пустой
   массив = подходит любой теме), `revenue_per_click` (оценка, ₽), status.
   CRUD через CLI (`offer-add`, `offer-list`, `offer-set-status`).

2. **Атрибуция**: `tracked_links.offer_id` — клик по короткой ссылке
   атрибуцируется офферу; расчётный доход = clicks × revenue_per_click,
   виден в дашборде по офферам и по постам.

3. **Выбор оффера** при выпадении CTA `affiliate_link` (генерация, prod):
   Thompson sampling по существующим bandit-таблицам (dimension `offer`,
   arm = slug), успех = «пост с оффером получил ≥1 клик за 48h».
   Выбор — argmax ожидаемого дохода `θ·revenue_per_click`; пока ни у одного
   оффера нет известной ставки, выбирается по чистому θ. Исходы собирает
   `update_offer_outcomes()` в bandit-кроне (идемпотентно через
   bandit_observations).

4. **Graceful fallback** (AC issue #38): пустой каталог, недоступный PG,
   невыставленный TRACKING_BASE_URL или title, не прошедший promotional-gate,
   → CTA пересэмплируется из остальных вариантов; пост выходит без оффера.
   Выбранный оффер логируется в `summary.offer_slug` (попадает в
   post_features.policy_blob через engagement collector) и в structlog.

## Последствия

- Доход стал данными: дашборд показывает расчётный доход, бандит учится
  на кликах по офферам, а клики уже входят в композитный reward (ADR-0010).
- Взвешивание кликов реальной ставкой оффера внутри reward отложено до
  накопления фактических данных о доходе (нужна калибровка ставок офферов).
- Ставка `revenue_per_click` — ручная оценка; фактический доход партнёрок
  надо периодически сверять и обновлять через `offer-add` (upsert).
