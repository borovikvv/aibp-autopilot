# ADR-0008: Байесовский (bootstrap) критерий решений вместо жёстких p-value порогов

**Status:** Accepted
**Date:** 2026-07-06
**Related:** ADR-0007 (interleaving), issue #17

## Контекст

Старый критерий promote: Welch's t-test `p < 0.05` **и** Cohen's `d > 0.3`
**и** `+10%` улучшение. При 2 постах/день и 7-дневном окне выборка n≈14 на
группу. Мощность такого теста на реалистичных эффектах (+10–30% engagement
rate при высокой дисперсии постов) близка к нулю: почти каждый эксперимент
заканчивался `insufficient_data` или `no_significant_improvement`. Фактически
автопилот не мог принять ни одного решения — self-learning выродился в
генератор отчётов.

## Решение

Комбинация трёх мер из issue #17:

1. **Окно эксперимента 7 → 14 дней** (`safety.experiment_window_days: 14`).
   Удваивает n до ≈28 на группу без изменения темпа постинга.

2. **Bootstrap-оценка `P(variant > control)`** вместо t-test + Cohen's d.
   4000 ресэмплов средних обеих групп; promote при
   `P(variant > control) >= 0.95` **и** относительном эффекте `>= 5%`
   (`safety.promote_probability`, `safety.min_effect_pct`). Bootstrap не
   предполагает нормальность (распределение engagement rate постов скошено)
   и даёт прямо интерпретируемую величину — вероятность того, что вариант
   лучше. Порог 0.95 держит false-positive rate ≈5% на нулевом эффекте
   (проверяется регресс-тестом на 200 синтетических экспериментах), фильтр
   минимального эффекта отсекает продвижение «статистически достоверных»,
   но бесполезных изменений.

3. **Реалистичный целевой эффект в гипотезах pattern miner**: пример в
   промпте изменён с «+10%» на «+25%» — гипотезы с ожидаемым эффектом ниже
   уровня обнаружения не имеют смысла.

Семантика полей `experiments_log` сохранена структурно, но переозначена:
`p_value` теперь хранит `P(variant > control)` (0..1, больше = лучше),
`effect_size` — относительный эффект (доля, 0.30 = +30%), а не Cohen's d.

## Калибровка на реальных данных

Пороги 0.95 / 5% — априорные, подтверждены только симуляцией (см.
`tests/unit/test_decision_engine.py::test_false_positive_rate_under_5_percent`).
После первых 4–6 недель работы interleaving-экспериментов нужно пересчитать
фактическую дисперсию engagement rate по `post_features`/`engagement_metrics`
(горизонт 48ч, ADR/issue #14) и при необходимости скорректировать
`min_effect_pct` вверх, если дисперсия окажется выше принятой в симуляции
(σ/μ = 0.2). Это отражено в policy как обычные safety-параметры — менять их
может только человек, автопилоту секция safety недоступна.

## Последствия

- Автопилот сможет принимать решения на реальных объёмах данных канала.
- Цена: чувствительность к малым эффектам (<5%) сознательно отброшена.
- Дашборд теперь показывает `P(shadow>control)` в колонке p-value —
  интерпретация «ближе к 1 — лучше», а не «меньше 0.05 — лучше».
