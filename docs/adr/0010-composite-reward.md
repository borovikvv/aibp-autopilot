# ADR-0010: Композитная reward-функция вместо views/subscribers

**Status:** Accepted
**Date:** 2026-07-07
**Related:** ADR-0008 (bootstrap criterion), issue #37, issue #15 (click tracking)

## Контекст

Decision engine и бандит оптимизировали `views / subscribers` — метрику
удержания текущей аудитории. Цели канала другие: рост подписчиков и
монетизация. Система самообучалась делать посты, которые смотрят существующие
подписчики, но не получала никакого сигнала о том, что приносит новых
подписчиков (Δsubs) или клики по трекинг-ссылкам (данные redirect-сервиса,
issue #15, вообще не входили в reward). Оптимизация сходилась не к целям.

## Решение

Единая композитная reward-функция (`src/aibp/self_learning/reward.py`),
используемая и decision engine, и бандитом:

```
reward = (w_views·views + w_forwards·forwards + w_clicks·clicks + w_subs·Δsubs)
         / subscribers_at
```

Все компоненты — на фиксированном 48h-горизонте (ADR-0004/issue #14),
нормировка на подписчиков сохраняет сравнимость постов при росте канала.

1. **forwards** — главный органический канал роста и сигнал для алгоритма
   рекомендаций Telegram; уже собирались engagement collector'ом, но не
   использовались.
2. **clicks** — клики по трекинг-ссылкам поста в окне 48h из PostgreSQL
   (`link_clicks`). Если PG недоступен — компонент деградирует до 0 для
   *всех* постов с warning в логе: обе группы эксперимента теряют его
   одинаково, сравнение остаётся валидным.
3. **Δsubs** — прирост подписчиков, атрибуцированный посту. Строится
   глобальный ряд `subscribers_at` из всех снапшотов engagement_metrics
   (main-канал), линейная интерполяция. Окно атрибуции:
   `[posted_at, posted_at + subs_attribution_hours]`, обрезается временем
   следующего main-поста — при 2 постах/день два поста никогда не делят
   один прирост. Ночной прирост после вечернего поста засчитывается
   вечернему (его окно дотягивается до утреннего). Отрицательный Δsubs
   (отписки) сохраняется как сигнал.

Веса — эвристические оценки ценности (форвард ≈ 25 просмотров, подписчик ≈ 50)
в `policy.yaml → safety.reward_weights`; safety-блок автопилотом не
модифицируется. Уточнять вручную по мере накопления данных о фактическом
CPS и доходе на клик (issue #38, #39).

## Что осталось на старой метрике

- **auto_rollback / anomaly detection** — намеренно: rollback-триггеры
  должны реагировать на просадку охвата как таковую, независимо от весов
  reward; менее шумный предохранитель.
- **compute_engagement_rates** сохранена как legacy-метрика для
  наблюдаемости и сравнения.
- **pattern_miner** — переход на reward отложен: гипотезы майнятся по
  engagement, но проверяются экспериментом уже на reward, так что ошибка
  майнера не промоутится.

## Последствия

- Бандит и эксперименты начинают продвигать паттерны, дающие подписчиков и
  клики, а не только просмотры.
- Δsubs шумный (дневные колебания канала целиком приписываются постам);
  при малых каналах компонент может доминировать в дисперсии. Компенсируется
  bootstrap-критерием (ADR-0008) и умеренным весом.
- История bandit_observations, накопленная на старой метрике, не
  пересчитывается: посте́риоры дообучатся на новых наблюдениях.
- Дашборд показывает декомпозицию reward по компонентам для каждого поста
  (секция «Composite Reward»).
